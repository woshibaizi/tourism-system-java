# 个性化旅游系统 — 功能设计与创新改进方案

> 2026-05-11，基于全量源码审查与功能目录分析

---

## 一、现有功能的设计缺陷

### 1.1 ChatAgent 工具注册了但从未被 LLM 调用

**现状**：`chat_agent.py` 注册了 3 个工具（`search_places`、`get_place_detail`、`get_hot_places`），但在 `_process_with_llm()` 中：

```python
# chat_agent.py:127 — 直接调 LLM chat，没有传 tools 参数
content = client.chat(msgs)  
# 工具定义通过 get_tools() 暴露了，但从未进入 LLM 请求体
```

LLM 收到的请求里根本没有 tools 列表，自然无法触发 function calling。用户问"推荐附近有什么好吃的"，LLM 只能凭训练数据编造回复，永远调不到真实数据。

**修复方向**：在 `llm_client.chat()` 增加 `tools` 参数支持，ChatAgent 将 `registry.get_definitions()` 传入，LLM 返回 tool_call 时由 `registry.dispatch()` 执行并将结果回传 LLM 做最终回复。

---

### 1.2 路线规划工具是硬编码占位

**现状**：`agent-service/app/tools/route_planner.py`：

```python
def build_route_outline(requirement: str) -> dict:
    return {
        "summary": "已生成一个可继续细化的轻量路线草案",
        "estimated_minutes": 180 if "半天" in requirement else 360,
        # 没有调用任何 Java 后端接口
    }
```

Agent 的路线规划能力完全没有接到 Java 的 `NavigationController`（A*/Dijkstra/TSP 均已实现且可用）。

**修复方向**：`route_planner.py` 调用 `tourism_api_client` 的 Java 后端导航接口，返回真实的路径、距离和时间。

---

### 1.3 推荐系统不考虑上下文

**现状**：`RecommendationAlgorithm` 是纯 TF-IDF + 余弦相似度，输入只有 item 的关键词向量和 user 的评分向量。以下信息完全被忽略：

- **时间**：早上推荐咖啡馆 vs 晚上推荐酒吧
- **季节**：春天推荐赏花点 vs 冬天推荐温泉
- **天气**：晴天推荐户外 vs 雨天推荐室内
- **用户当前位置**：推荐附近 vs 推荐全城
- **用户近期行为**：刚去过的地方不再重复推荐

**修复方向**：在推荐分中加入上下文权重层——时间分、季节分、距离分、新鲜度分，与内容相似度分加权求和。

---

### 1.4 统计页面只是 7 个数字

**现状**：`StatsController.overview()` 返回：

```json
{"places": 201, "buildings": 63, "facilities": 204, "foods": 40, "diaries": 14, "users": 10, "roads": 483}
```

没有任何趋势、排行、个人维度或时间维度数据。

**修复方向**：增加用户个人统计（我的足迹、今年去过的地方数、总步行距离、写过多少日记、平均评分），系统级趋势（本月热门目的地、热门菜系排行）。

---

### 1.5 首页千人一面，无个性化

**现状**：`HomePage` 加载固定数据——热门场所、推荐日记、系统统计。所有用户看到完全相同的首页，无论他们的旅行偏好是什么。

**修复方向**：基于 `user_behavior` 表的历史数据为每个用户生成个性化首页——"猜你喜欢"、"继续探索"（最近浏览过的地方的同类推荐）、"你的旅行回顾"。

---

### 1.6 美食与场所割裂

**现状**：美食搜索（`FoodSearchPage`）和地点搜索（`LocationSearchPage`/`PlacesPage`）是独立页面。用户看一个景点时看不到附近美食，看美食时也看不到附近有什么可逛的。

这是一个典型的"数据有但体验没有"的问题——后端有关联查询能力（`/api/foods/place/{placeId}`、`/api/facilities/place/{placeId}`），但前端没有把它们组织成连贯体验。

**修复方向**：在场所详情页直接展示"附近美食"和"附近设施"卡片，在美食详情页展示"所在场所"和"附近景点"。

---

### 1.7 日记写作无辅助

**现状**：日记管理页是纯粹的手动表单填写（标题 + 正文 + 图片链接）。Agent 的 `DiaryAgent` 可以做自动生成，但两者之间没有连接——用户要么完全手动写，要么完全让 AI 写，没有中间地带。

**修复方向**：在日记编辑器中增加 AI 辅助按钮——"帮我润色"、"生成标题"、"从图片提取描述"、"续写"。让用户主导、AI 辅助，而非二选一。

## 二、创新功能提案

以下提案的核心原则：**可实现性高**（依赖现有数据和算法，不引入全新的技术栈）、**契合项目定位**（个性化旅游）、**有展示价值**（能作为项目亮点）。

---

### 2.1 智能一日游规划器 ★★★★★

**一句话**：输入"我想在杭州玩一天，预算 500，喜欢拍照和美食，不要太累"，输出带时间轴的完整行程。

**为什么现在能做**：
- 场所数据库有 201 个地点，含类型、坐标、评分
- 美食数据库有 40 个菜品，含菜系、价格、人气
- 路径规划算法（A*/Dijkstra/TSP）均已实现
- Agent 有意图识别和 LLM 调用能力
- 缺失的只是"把这三者编排到一起"的编排层

**实现路径**：

```
用户输入自然语言需求
  → Dispatcher 识别意图 + 提取 slot(地点/时长/预算/兴趣/节奏)
  → 调用 POST /api/places/recommend 获取候选景点（按兴趣加权）
  → 调用 POST /api/foods 获取候选餐饮（按预算+菜系偏好）
  → 调用 POST /api/navigation/multi-destination 计算最优游览顺序
  → 编排成时间轴：09:00 出发 → 09:30 景点A → 11:30 午餐 → ...
  → 前端渲染为可交互的行程卡片（可拖拽调整顺序、替换地点）
```

**前端展示**：一条垂直时间轴，每段显示地点卡片 + 交通方式 + 耗时 + 花费估算。支持拖拽调整顺序后自动重算路线。

**创新点**：这不是简单的路线规划（那是 TSP），而是带多约束（时间预算、餐饮预算、节奏偏好）的行程编排。国内旅游类 app 很少做到这个粒度。

---

### 2.2 旅行人格分析 ★★★★★

**一句话**：基于用户的历史行为自动生成"旅行人格"画像，并影响推荐策略。

**旅行人格类型**（示例）：

| 人格 | 特征 | 推荐策略 |
|------|------|----------|
| 美食猎人 | 评分偏向美食、浏览大量餐厅 | 优先推荐高评分美食 + 新开餐厅 |
| 摄影漫游者 | 偏好景区、评分含"拍照""好看" | 推荐拍照点、日落黄金时刻路线 |
| 文化探索者 | 偏好博物馆、历史建筑 | 推荐文化类场所 + 相关日记 |
| 高效打卡族 | 一日多目标、路程紧凑 | 推荐高密度区域 + 最优路线 |
| 悠闲度假派 | 节奏慢、评分高但数量少 | 推荐安静场所 + 咖啡馆 + 轻松路线 |

**实现路径**：
1. 从 `user_behavior` 表聚合用户行为特征向量（浏览类型分布、评分均值/方差、活跃时段、常用关键词）
2. 用规则引擎（非 ML）将特征向量映射到人格类型
3. 在推荐算法中按人格加权不同类型 item 的得分
4. 在首页展示人格卡片 + 个性化推荐理由

**前端展示**：首页顶部一个可分享的"旅行人格卡片"（含人格名称、描述、统计数据、推荐语），可截图分享。

**创新点**：Spotify Wrapped 式的个性化年度/季度回顾在旅游领域的应用。国内尚无旅游 app 做类似的用户画像可视化。

---

### 2.3 旅行回忆自动生成 ★★★★

**一句话**：用户完成一次旅行后（连续的日记 + 浏览记录），一键生成图文并茂的"旅行回忆"。

**为什么现在能做**：
- 有日记数据（标题、正文、图片、目的地）
- 有 AIGC 能力（图片处理、GIF 生成）
- 有用户行为数据（浏览过哪些地方）
- 缺的是"聚合 + 可视化"层

**实现路径**：
1. 用户选择一次旅行的时间范围（或系统自动识别日记的时间聚类）
2. 系统聚合：日记摘要、照片、访问地点、总步行距离（从导航记录估算）、美食清单、天气标签
3. 用 DiaryAgent 生成一段"旅行总结"文字（如"这次杭州之行你走过了 12 公里..."）
4. 输出为一张可分享的卡片图（类似 Strava 运动总结）或一个 GIF 动画

**前端展示**：DiariesPage 增加"生成旅行回忆"入口，选择日期范围后展示预览，确认后生成。

**创新点**：把离散的旅行数据转化为有情感价值的回忆产品。类似 Apple Photos 的"回忆"功能，但面向旅行场景。

---

### 2.4 "此刻出发"场景化即时推荐 ★★★★

**一句话**：根据当前时间、位置、天气，推荐"现在最适合做的事"。

**场景举例**：

| 触发条件 | 推荐内容 |
|----------|----------|
| 11:30，距离食堂 500m 内 | "午饭时间到了，第三食堂的麻辣香锅评分 4.8★，步行 4 分钟" |
| 17:30，晴天，靠近湖边 | "日落前一小时，湖心亭是最佳拍照点，步行 8 分钟" |
| 14:00，下雨 | "下雨天适合室内——图书馆咖啡厅的提拉米苏很受欢迎" |
| 周末上午，用户喜欢拍照 | "周末早晨光线好，植物园人少适合拍照，评分 4.6★" |
| 用户首次打开 app | "欢迎来到杭州！为你推荐 3 个必去地点→" |

**实现路径**：
1. 前端获取用户位置（浏览器定位）和时间
2. 后端根据时间/位置/天气/用户偏好生成推荐列表
3. 推荐算法新增"场景相关性"权重维度
4. 在首页顶部以卡片轮播形式展示，可下拉刷新

**创新点**：把推荐从"搜索式"变为"推送式"，降低用户决策成本。类似 Google Maps 的"Explore" tab 但更个性化。

---

### 2.5 校园 AR 语音导览 ★★★

**一句话**：在校园导航的基础上，对途经的地标建筑自动播放语音介绍。

**为什么现在能做**：
- 有 63 个建筑物数据（名称、坐标、描述）
- 有坐标转换能力（WGS84 → GCJ-02）
- 有校内导航功能（`CampusNavigationPage`）
- 有高德地图 SDK 集成

**实现路径**：
1. 为 `SpotBuilding` 实体增加 `audio_description` 字段（或用 TTS 从文字描述生成）
2. 在导航过程中，当用户位置靠近某个建筑物 50m 范围内时自动弹出信息卡片 + 播放语音
3. 语音使用浏览器内置 SpeechSynthesis API（零成本，无需后端 TTS 服务）

**前端实现**：在 `AMapView` 组件中增加建筑物 proximity 检测，当用户 marker 接近建筑物 marker 时触发语音播报。

**创新点**：把导航从"怎么走"升级为"走的过程中了解这个地方"。成本极低（浏览器 TTS + 已有数据），但体验差异巨大。

---

### 2.6 多人协作行程规划 ★★★

**一句话**：朋友间共享一个行程，各自添加想去的地方，投票决定优先级，系统自动计算最优路线。

**实现路径**：
1. 新增 `trip_plan` 表（id, creator_id, title, status, created_at）
2. 新增 `trip_plan_item` 表（plan_id, place_id, added_by, votes）
3. 用户创建行程后生成分享链接/邀请码
4. 参与者各自添加想去的地方、投票
5. 最终确认后调用 TSP 多目标路径规划

**前端展示**：行程详情页（参与者头像、地点列表 + 票数、地图预览），确认后展示完整路线时间轴。

**创新点**：解决"一群人出去玩不知道去哪"的决策困难。当前市场上多人行程规划产品体验普遍不好。

---

### 2.7 美食图谱 ★★

**一句话**：以可视化图谱展示美食之间的关系——什么菜系在哪个食堂、哪家店和哪家店口味相近、用户自己的口味偏好地图。

**实现路径**：
1. 给 `spot_food` 实体增加口味标签（辣/甜/咸/清淡/酸）
2. 用已有的相似度计算（`cosineSimilarity`）构建美食-美食相似矩阵
3. 前端用简单的力导向图或雷达图展示

**前端展示**：个人口味雷达图 + "你可能也喜欢"推荐列表。

## 三、功能组合建议：一个完整的用户旅程

把以上创新点串联成一个演示场景，作为项目展示的"黄金路径"：

```
1. 用户打开 App → 首页展示"旅行人格卡片"（美食猎人）
   + "此刻推荐"：午饭时间的食堂推荐

2. 点击"智能行程规划" → 输入"周末在杭州玩一天，喜欢拍照和美食"
   → 系统生成完整时间轴行程，包含 4 个景点 + 2 顿饭 + 步行路线

3. 出发当天 → 打开校园导航，选择行程中的第一个地点
   → 导航过程中途经图书馆自动播放语音："这是学校图书馆，建于1955年..."

4. 旅行结束后 → 收到通知"你的一天旅行回忆已生成"
   → 查看自动生成的旅行回顾卡片，包含走过的路、吃过的美食、拍的照片
   → 一键分享到朋友圈

5. 朋友看到分享 → 通过邀请码加入"下次一起去"协作行程
   → 各自添加想去的地方 → 投票 → 系统生成最优路线
```

这个流程串起了：旅行人格 → 智能规划 → AR 语音导览 → 旅行回忆 → 协作行程，五个创新功能全部基于现有数据实现。

## 五、关键设计决策

### 决策 1：智能行程规划器的粒度

**选轻量方案**：先做"半日/一日"的单天行程规划，不做多日酒店+机票预订。理由：多日行程需要大量外部数据（酒店、火车票、门票价格），超出当前数据能力。单日行程完全基于现有场所/美食/道路数据即可实现，闭环可控。

### 决策 2：旅行人格的计算方式

**选规则引擎，不选 ML 模型**。理由：用户量小（当前 10 个测试用户），ML 模型无意义。用规则引擎（如"浏览美食占比 > 40% → 美食猎人"）可解释性强、开发快、足够准确。后续数据积累后可替换为轻量分类模型。

### 决策 3：语音导览的实现方式

**选浏览器 TTS，不选后端 TTS 服务**。理由：零成本、零延迟、无需依赖外部 API。用 `SpeechSynthesis` API 朗读已有的文字描述即可，校园场景下文字描述提前写好质量有保证。

### 决策 4：旅行回忆的输出格式

**首选可分享图片卡片，次选 GIF 动画**。理由：图片卡片便于社交分享（微信/小红书），已有 AIGC GIF 生成能力可作为增强。用 HTML → Canvas → PNG 生成卡片图，或使用 `html2canvas` 库。
