# SystemCraft 课程展示亮点整理

## 1. 项目一句话定位

**SystemCraft 不是一个“让 AI 直接生成结果”的工具，而是一个把软件工程全过程拆解出来、可视化出来、并进一步转化为学习过程的多 Agent 教学平台。**

如果要用更适合课堂展示的一句话来介绍：

> 我做的不是普通的 AI 生成器，而是一个用多 Agent 模拟软件团队协作、展示软件工程全过程、并支持学习复盘的智能工作台。

---

## 2. 展示时最值得强调的核心价值

很多 AI 项目只会展示“最后生成了什么”，但 SystemCraft 更关注：

1. **软件工程过程是否被清晰展示**
2. **Agent 是否真的承担不同角色**
3. **系统是否具备工程闭环，而不是一次性生成**
4. **项目结果能否继续转化为学习资产**

所以这个项目最大的亮点，不是简单地“用了多个 Agent”，而是把 Agent 和软件工程流程、学习流程、展示流程真正结合了起来。

---

## 3. 课程展示时可以重点讲的项目优点

## 3.1 多 Agent 设计不是表面分工，而是真正映射软件工程角色

SystemCraft 把一个软件项目拆成了多个明确角色：

- PM Agent：负责需求分析
- Arch Agent：负责架构设计
- Dev Agent：负责开发实现
- QA Agent：负责测试评审
- Mentor Agent：负责导师总结
- Learning Agent：负责项目完成后的追问与知识沉淀

这意味着系统不是“同一个模型换几个名字”，而是在用多角色协作的方式模拟真实软件团队。

这个点在课堂展示时很有优势，因为它能自然引出一个更高级的话题：

**Agent 的价值不只是生成内容，而是承担复杂流程中的职责。**

---

## 3.2 项目强调“过程可见”，不是只给最终结果

很多同类 AI 项目只展示最终生成物，比如一份文档、一段代码或一个方案。

而 SystemCraft 展示的是完整过程：

- 需求文档
- 架构设计
- 开发产物
- 测试报告
- 导师总结
- 知识卡片

并且这些内容不是静态堆在一起，而是和 Agent 的执行顺序、消息流、流程图联动展示。

这使得项目特别适合教学和答辩场景，因为老师看到的不是一份“黑盒结果”，而是一条完整的工程链路。

---

## 3.3 QA 回流机制体现了真正的软件工程闭环

SystemCraft 的关键亮点之一，是 **QA Agent 不只是输出测试结果，而是承担流程控制职责**。

流程不是简单的：

`需求 -> 架构 -> 开发 -> 结束`

而是：

`需求 -> 架构 -> 开发 -> QA -> 如果发现问题则回到开发 -> 修复后再测试 -> 最后进入导师总结`

这个设计非常重要，因为它体现了真实软件工程中的返工逻辑。

在课程展示时，这一点可以重点讲成：

**我的系统不是线性生成流水线，而是带质量反馈回路的工程闭环。**

这比“我用了多 Agent”更能体现你对软件工程和 Agent workflow 的理解。

---

## 3.4 LangGraph 的使用体现了“从原型验证到工程取舍”的思考

这个项目有一个很适合展示的方法论亮点：

- 前期通过 LangGraph 验证多 Agent 图式工作流是否成立
- 后期在正式系统中演进为更可控的自定义编排层

这样做不是因为 LangGraph 不好，而是因为正式教学系统除了流程执行，还需要：

- 项目暂停与恢复
- 状态持久化
- WebSocket 实时同步
- 前端过程可视化
- 项目迭代记录

所以你可以把这一点讲成：

**我不是为了“用了框架”而用框架，而是在理解 LangGraph 的图式工作流思想后，结合产品需求做了工程化取舍。**

这个表述会比单纯说“我用了 LangGraph”更成熟。

---

## 3.5 共享状态 State 设计让多 Agent 协作可解释、可扩展

SystemCraft 的多 Agent 协作不是靠混乱的 prompt 传递，而是靠统一状态对象驱动。

状态中包括：

- 用户输入
- 项目名称
- 需求文档
- 架构文档
- 代码产物
- 测试报告
- 导师总结
- 知识卡片
- 当前阶段
- 是否存在问题
- 迭代次数
- Agent 消息流

这种设计的优点是：

1. 每个 Agent 只读前序结果、只写自己负责的产物
2. 输入输出边界清楚，便于解释
3. 前端很容易把状态映射成流程图和产物视图
4. 后续扩展更多 Agent 或更多阶段时结构仍然清晰

你在展示时可以用一句很有力量的话来总结：

**我的多 Agent 系统不是靠“多次调用模型”拼出来的，而是靠共享状态组织起来的。**

---

## 3.6 前端把 Agent 过程真正做成了“可视化工作台”

SystemCraft 的前端不是普通聊天界面，而是一个三栏工作台：

1. **左侧流程图**：展示当前 Agent 所在阶段和整体工作流
2. **中间消息流**：展示不同 Agent 的协作内容和阶段消息
3. **右侧产物面板**：展示需求、架构、代码、测试、总结等实际交付物

这样做的价值是：

- 用户能看到“系统现在在做什么”
- 用户能看到“是谁在做”
- 用户能看到“产出了什么”

这使得整个项目非常适合课程展示，因为老师可以在短时间内理解系统结构、执行过程和最终结果。

---

## 3.7 项目不仅“交付项目”，还“教会项目”

这是 SystemCraft 最有教学特色的一点。

项目完成之后，系统并不会停止，而是继续进入学习阶段：

- Mentor Agent 输出复盘总结
- Learning Agent 接收学生追问
- 追问结果可以沉淀成知识卡片
- 系统支持思考题和学习进度记录

这意味着 SystemCraft 不只是一个生成系统，更是一个学习系统。

你可以在展示时强调：

**很多 AI 项目解决“做出来”的问题，而我的项目还在解决“看懂为什么这样做”的问题。**

这会让老师很容易感受到项目的教学价值。

---

## 3.8 追踪矩阵让需求、架构、实现、测试之间的关系被显式展示

SystemCraft 支持从不同阶段产物中提取信息，构造“需求 -> 架构 -> 实现 -> 测试”的追踪视图。

虽然目前还是启发式抽取，但它已经很好地服务了教学场景：

- 学生能看到需求如何落到架构
- 学生能看到架构如何影响实现
- 学生能看到测试如何验证实现

这相当于把软件工程里常说的“可追踪性”做成了直观可见的界面。

这个亮点尤其适合老师喜欢“工程思维”的场景。

---

## 3.9 工程化程度高于普通 Demo

SystemCraft 不只是一个实验脚本，而是一个较完整的前后端应用。

已经具备的工程化能力包括：

- FastAPI 后端接口
- WebSocket 实时推送
- React 前端工作台
- SQLite 持久化
- 账号与会话管理
- 项目历史管理
- 项目暂停与恢复
- 追问与知识卡片保存
- 迭代记录
- Markdown 与 HTML 导出
- Windows 一键启动脚本

这意味着项目不是停留在“能跑起来”，而是已经进入“能演示、能管理、能复盘”的应用形态。

---

## 4. Agent 创新角度怎么讲最有说服力

如果老师特别关注 agent 创新，你可以把项目分成两个层面来讲。

## 4.1 系统层创新

系统层面，SystemCraft 做的是：

- 用多 Agent 映射软件工程角色
- 用共享状态驱动协作
- 用 QA 回流构造工程闭环
- 用实时工作台把流程显式化
- 用导师和学习模块把结果转化为学习资产

也就是说，Agent 在这个系统里不是“多个调用”，而是“流程中的角色和节点”。

## 4.2 开发过程创新

如果展示时你要讲到自己的开发方法，可以这样表达：

> 这个项目不只是研究多 Agent 产品架构，我在开发过程中也在探索如何通过 skills 和 harness 约束 agent 的行为，让 agent 更稳定地参与真实开发。

这样你就把自己的创新讲成了双层结构：

1. **产品层**：SystemCraft 本身是一个多 Agent 教学系统
2. **方法层**：开发这个系统的过程也体现了 agent engineering 的思路

这个说法很适合老师喜欢“创新方法论”的场景。

---

## 5. 为什么这个项目比普通 AI 项目更适合课程展示

SystemCraft 很适合课程展示，原因主要有四个：

1. **过程可见**：不是黑盒生成，老师容易理解
2. **角色鲜明**：多 Agent 分工天然适合讲解
3. **结果完整**：既有文档、架构、代码、测试，也有总结和学习成果
4. **展示性强**：工作台、成果页、导出页都适合现场演示

简单说，就是这个项目既有“技术味”，也有“展示味”，非常适合答辩和课堂汇报。

## 6. 展示时可以直接用的表达句子

### 6.1 开场一句话

> 我做的不是一个简单的 AI 生成器，而是一个用多 Agent 模拟软件团队协作、展示软件工程全过程、并支持学习复盘的教学平台。

### 6.2 讲核心创新时

> 这个项目的创新点不在于“Agent 数量多”，而在于我让 Agent 变成了软件工程流程里的真实角色，并通过状态和回流机制形成了工程闭环。

### 6.3 讲 QA 回流时

> 我的系统不是线性生成流水线，而是带测试反馈回路的流程系统，这一点更接近真实的软件工程实践。

### 6.4 讲教学价值时

> 很多 AI 系统解决的是“做出来”的问题，而我这个项目还在解决“为什么这样做”和“学生如何学会”的问题。

### 6.5 讲 LangGraph 时

> 我前期用 LangGraph 验证多 Agent 图式流程，后期为了适应暂停恢复、持久化和可视化等产品需求，把它演进为更可控的编排层，这是一次从原型到工程系统的自然演进。

---

## 7. 如果老师问“你这个项目最大的亮点是什么”

你可以用下面这段回答：

> 我觉得最大的亮点不是用了多 Agent，而是把 Agent 真正放进了软件工程流程里。  
> 它们各自承担需求、架构、开发、测试、复盘这些职责，并通过共享状态和 QA 回流形成闭环。  
> 最后系统还把项目结果继续转化成知识卡片、追问问答和学习进度。  
> 所以这个项目既是一个多 Agent 系统，也是一个软件工程教学系统。

---

## 8. 适合你的展示主线

为了让展示更顺，你可以按下面这条主线来讲：

1. **先讲问题**：普通 AI 项目只给结果，不展示过程
2. **再讲思路**：用多 Agent 模拟软件团队协作
3. **再讲架构**：共享状态、阶段流转、QA 回流
4. **再讲界面**：流程图、消息流、产物预览
5. **再讲教学价值**：导师总结、追问、知识卡片、学习闭环
6. **最后总结**：SystemCraft 让软件工程从“黑盒生成”变成“可观察、可解释、可学习”的过程

---

## 9. 最后总结

SystemCraft 的真正价值，可以概括成一句话：

**它不是在用 AI 替代软件工程过程，而是在用多 Agent 把软件工程过程展示出来、组织起来，并进一步转化为学习过程。**

如果从课程展示角度评价，这个项目的优势非常明显：

- 有技术深度
- 有架构思考
- 有工程闭环
- 有界面展示
- 有教学价值
- 有 agent 创新叙事

这也是它最适合在课堂上“讲出层次、讲出方法、讲出亮点”的原因。
