# 写游记 AI 接入 & 选项精简 — 优化方案

> 编写日期：2026-06-03
> 涉及模块：Web前端 (React)、Agent服务 (Python)

---

## 一、现状分析

### 1.1 当前有两条完全割裂的写作路径

| 路径 | 入口 | AI参与 | 字段数 | 体验 |
|------|------|--------|--------|------|
| **写游记** (手动) | DiariesPage / MyDiariesPage 的"写游记"按钮 | ❌ 零AI | 6个字段 | 纯手工填写，字段冗杂 |
| **生成游记** (AI) | PersonalTravelAssistantPage 聊天中触发 DiaryModal | ✅ 5阶段流水线 | 风格选择 + 描述 + 图片 | 功能完善但入口深 |

### 1.2 手动"写游记"的具体问题

**Web端 (DiariesPage.jsx 285-337行)** 当前表单字段：

| 字段 | 类型 | 问题 |
|------|------|------|
| 标题 | 必填文本 | ✅ 合理 |
| 内容 | 必填文本区 | ✅ 合理（核心字段） |
| 目的地 | 自由文本 | ❌ 冗余——与"所属景区"下拉框功能重叠，且自由文本无数据关联 |
| 所属景区 | 下拉选择 | ⚠️ 有用但位置不佳，应该和目的地合并 |
| 图片 | 文件上传 | ⚠️ 无预览、无AI识图 |
| 视频 | 文件上传 | ❌ 使用率极低，增加表单复杂度 |

### 1.3 "生成游记"功能已非常完善

- 3种风格模式（预设/调参/自述）、6个预设模板
- 图片视觉理解 → 要素提取 → 正文 → 润色 → 持久化，5阶段流水线
- 支持换风格重新生成
- 进度轮询 + 结果编辑 + 一键发布
- 唯一的缺陷：**入口太深**，必须在聊天页面触发，不在常规写游记流程中

---

## 二、优化目标

1. **打通 AI**：让"写游记"也能用上 AI 能力，用户可选择是否使用 AI
2. **精简选项**：砍掉冗余字段，减少用户决策负担
3. **统一入口**：不再有两条割裂的路径，一个入口覆盖所有写游记场景
4. **渐进式 AI**：AI 是辅助而非替代——用户可以纯手动、半AI、全AI

---

## 三、优化方案

### 3.1 统一入口 & 模式选择

将"写游记"按钮点击后的弹窗改为**模式选择 + 写作区**的一体化设计：

```
┌──────────────────────────────────────┐
│  写游记                          [X] │
├──────────────────────────────────────┤
│                                      │
│  ○ AI 辅助写作    ○ 快速记录         │  ← 顶部模式切换
│                                      │
├──────────────────────────────────────┤
│  （根据所选模式展示不同内容）          │
│                                      │
└──────────────────────────────────────┘
```

**两种模式：**

| 模式 | 场景 | 说明 |
|------|------|------|
| **AI 辅助写作** | 想要高质量、有风格的游记 | 复用现有 DiaryModal 的完整 AI 能力 |
| **快速记录** | 只想快速记几笔，不需要 AI | 精简到最少字段，但保留"AI润色"按钮作为可选增强 |

### 3.2 "快速记录"模式 — 选项精简

**砍掉方案：**

| 原字段 | 处理 | 理由 |
|--------|------|------|
| 标题 | **保留**，改为可选 + AI自动生成按钮 | 必填增加负担，可事后补充或AI生成 |
| 内容 | **保留**，核心字段 | 唯一真正必须的字段 |
| 目的地 | **删除** | 与"所属景区"重叠，自由文本无数据价值 |
| 所属景区 | **保留**，改为智能搜索选择器 | 从下拉改为搜索，数据量大时下拉不可用 |
| 图片 | **保留**，增加上传预览 | 保留但改善体验 |
| 视频 | **删除** | 极少使用，如需可后续加回 |
| 标签 | **新增**，AI自动生成 | 手动打标签体验差，AI来做 |

**精简后的快速记录表单（仅4项）：**

```
┌──────────────────────────────────────┐
│  标题  [________________] 🤖生成     │  ← 带AI生成标题按钮
│                                      │
│  内容  [________________]            │
│        [________________]  ✨AI润色  │  ← 内容区带AI润色按钮
│        [________________]            │
│                                      │
│  关联景区  [🔍 搜索景区...___]        │  ← 搜索式选择器，不再是下拉
│                                      │
│  图片  [📷 已选 3张] [+上传]         │  ← 带缩略图预览
│        [缩略图] [缩略图] [缩略图]     │
│                                      │
│  标签  #旅行 #美食  [🤖自动生成]      │  ← AI自动生成标签
│                                      │
│            [📝 发布游记]              │
└──────────────────────────────────────┘
```

### 3.3 AI 能力在"快速记录"中的嵌入方式

AI 不是全有或全无，而是**作为工具按钮嵌入**到手动流程中：

| AI 能力 | 触发方式 | 调用 | 效果 |
|---------|----------|------|------|
| 🤖 **生成标题** | 点击标题旁的🤖按钮 | 调用 Agent 根据内容生成3个候选标题 | 用户选择一个或手动改 |
| ✨ **AI润色** | 点击内容区下的✨按钮 | 调用 Agent 润色已有文本（修正错别字、优化表达） | 保留原意，提升可读性 |
| 🏷️ **自动标签** | 点击标签旁的🤖按钮 | 调用 Agent 从内容中提取关键词 | 3-5个标签，用户可增删 |
| 🖼️ **图生文** | 上传图片后出现"AI描述"按钮 | 调用 vision 模型分析图片 | 生成一段图片描述，可选插入正文 |
| 🎨 **套用风格** | 底部"风格"快捷选择 | 调用 Agent 将已写内容转为指定风格 | 快速切换文风 |

### 3.4 "AI 辅助写作"模式 — 接入现有能力

复用 `DiaryModal` 的完整流程，并做以下改进：

**保留的现有能力：**
- 三种风格模式：预设 / 调参 / 自述
- 6个预设风格模板
- 图片上传 + AI视觉理解
- 5阶段生成流水线
- 进度展示 + 结果编辑 + 一键发布

**新增改进：**
- 支持在 AI 生成结果上**手动修改后再发布**（已有，保留）
- 增加"先写草稿，AI润色"的半自动模式 —— 用户写几行要点，AI 扩展成完整游记
- 生成完成后可以一键切换到另一种风格重新生成（已有，保留）

### 3.5 两种模式的切换逻辑

```
用户点击"写游记"
      │
      ▼
┌─ 模式选择 ──────────────────────┐
│  ○ AI辅助写作    ○ 快速记录     │
│                                 │
│  默认记住用户上次选择            │
└─────────────────────────────────┘
      │                    │
      ▼                    ▼
┌──────────────┐   ┌──────────────┐
│ AI辅助写作    │   │ 快速记录      │
│ (完整AI流程)  │   │ (精简表单)    │
│              │   │              │
│ 上传图片     │   │ 输入内容     │
│ 选风格       │   │ 选景区       │
│ 补充描述     │   │ 传图片       │
│ → AI生成     │   │              │
│ → 编辑调整   │   │ [可选]AI润色  │
│ → 发布       │   │ [可选]AI标题  │
│              │   │ → 发布       │
└──────────────┘   └──────────────┘
```

- **快速记录 → AI辅助**：如果用户内容写了很多但不满意，可一键"升级为AI辅助"，已有内容作为草稿传入 AI 流程
- **AI辅助 → 快速记录**：如果 AI 生成的内容用户想大幅手动修改，可一键"转为手动编辑"

## 四、后端/Agent 改动

### 4.1 新增 Agent API

当前 Agent 只有完整的 `POST /agent/diary/generate`（5阶段流水线），需要新增轻量级 AI 能力接口：

| 接口 | 方法 | 说明 | 优先级 |
|------|------|------|--------|
| `/agent/diary/generate` | POST | 已有，全流程生成 | — |
| `/agent/diary/polish` | POST | **新增**：润色已有文本 | P0 |
| `/agent/diary/suggest-title` | POST | **新增**：根据内容生成标题候选 | P0 |
| `/agent/diary/extract-tags` | POST | **新增**：从内容提取标签 | P1 |
| `/agent/diary/describe-images` | POST | **新增**：图生文（已有能力，封装为独立接口） | P1 |
| `/agent/diary/apply-style` | POST | **新增**：将已有内容转换为指定风格 | P2 |

### 4.2 新增 Agent 方法

在 `DiaryAgent` 中新增：

```python
class DiaryAgent:
    # 已有
    async def process(self, request)        # 5阶段全流程
    async def regenerate(self, request)     # 换风格重新生成

    # 新增
    async def polish(self, content, style_hint=None)       # 纯文本润色
    async def suggest_title(self, content, count=3)        # 标题建议
    async def extract_tags(self, content, count=5)         # 标签提取
    async def describe_images(self, image_paths)           # 图生文（独立接口）
    async def apply_style(self, content, target_style)     # 风格转换
```

这些方法复用现有的 `build_style_prompt()` 和 LLM 调用基础设施，实现成本低。

### 4.3 前端 API 层新增

在 `frontend/src/services/api/agent.js` 中新增对应方法：

```javascript
export const polishDiary = (data) => apiPost('/agent/diary/polish', data);
export const suggestDiaryTitle = (data) => apiPost('/agent/diary/suggest-title', data);
export const extractDiaryTags = (data) => apiPost('/agent/diary/extract-tags', data);
export const describeDiaryImages = (data) => apiPost('/agent/diary/describe-images', data);
export const applyDiaryStyle = (data) => apiPost('/agent/diary/apply-style', data);
```

## 六、新旧对比

| 维度 | 现状 | 优化后 |
|------|------|--------|
| 入口数 | 2个分离入口（手动/AI） | 1个统一入口，内分2模式 |
| 手动表单字段 | 6字段（含冗余） | 3-4字段（精简） |
| AI 参与 | 0（手动路径） | 5个AI工具按钮可选使用 |
| 目的地 vs 景区 | 2个重叠字段 | 1个搜索式景区选择器 |
| 视频上传 | 有，极少用 | 移除（后续按需加回） |
| 标签 | 无/手填 | AI自动生成 + 手动编辑 |
| 标题 | 必填 | 可选，AI可生成 |
| 模式互通 | 不互通 | 可双向切换（升级/降级） |

## 七、风险 & 待确认

1. **AI 调用成本**：每次润色/标题/标签都是一次LLM调用。建议加防抖 + 每日调用上限提示
2. **响应延迟**：AI接口非流式时需2-5秒，需要合理的loading状态
3. **用户习惯**：砍掉"目的地"和"视频"字段可能影响已有用户，建议灰度或保留数据库字段（只隐藏UI）
4. **需确认**：是否所有 AI 功能都需要登录后才可用？（建议：是，避免匿名滥用）

---

## 八、总结

核心思路就一句话：**AI 不是替代用户写作，而是作为一组工具嵌入到写作流程中**。

- 用户想纯手动 → 快速记录，3个字段搞定
- 用户想AI帮忙 → 点润色/标题/标签按钮，哪里需要点哪里
- 用户想AI全写 → AI辅助模式，选风格传图片，一键生成

三个层次满足不同用户、不同场景的写作需求，没有冗余选项，没有割裂体验。
