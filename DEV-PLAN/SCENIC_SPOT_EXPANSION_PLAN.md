# 景区扩展方案：100 → 200 景区

## 一、现状分析

### 1.1 数据现状

| 维度 | 当前值 | 目标值 |
|------|--------|--------|
| 场所总数 | 201 | ~301 |
| 校园 (`校园`) | 101 条 | 101 条（不变） |
| 景区 (`景区`) | 100 条 | ~200 条 |
| 景区 ID 范围 | place_102 ~ place_201 | place_202 ~ place_301 |

### 1.2 SpotPlace 数据模型（18 个字段）

```
id               VARCHAR(32)    主键，如 "place_202"
name             VARCHAR(100)   景区名称
type             VARCHAR(20)    类型："景区"
keywords         JSON           关键词数组，如 ["故宫","北京","皇家宫殿"]
features         JSON           特色标签数组，如 ["世界文化遗产","明清皇宫","皇家建筑"]
rating           DECIMAL(3,1)   综合评分 0.0~5.0
rating_count     INT            评分人数
click_count      INT            浏览量/热度
lat              DECIMAL(10,7)  纬度（WGS84）
lng              DECIMAL(10,7)  经度（WGS84）
address          VARCHAR(255)   详细地址
open_time        VARCHAR(200)   开放时间
image            VARCHAR(500)   封面图片路径
description      TEXT           简要描述（~100字）
detail_description TEXT         详情页描述（~300-500字）
deleted          TINYINT(1)     软删除标记
created_at       DATETIME       创建时间（自动）
updated_at       DATETIME       更新时间（自动）
```

### 1.3 关联数据说明

- **spot_building**（建筑物）：现有景区基本无关联建筑，可暂不扩展
- **spot_facility**（设施）：现有景区基本无关联设施，可暂不扩展
- **spot_food**（美食）：现有景区基本无关联美食，可暂不扩展
- **spot_road_edge**（道路）：不依赖景区数据

**结论**：新增景区只需确保 spot_place 表数据完整，不需要关联扩展附属表。

### 1.4 API 接口分析

所有 API 都是通用设计，基于动态查询，**无硬编码的景区 ID 或数量限制**：

| 端点 | 兼容性 |
|------|--------|
| `GET /api/places` | ✅ 分页查询，自动包含新数据 |
| `GET /api/places/{id}` | ✅ 按 ID 查询，通用逻辑 |
| `GET /api/places/search` | ✅ LIKE 模糊搜索，通用逻辑 |
| `GET /api/places/hot` | ✅ 按 click_count 排序 |
| `GET /api/places/top-rated` | ✅ 按 rating 排序 |
| `GET /api/places/type/{type}` | ✅ 按类型过滤 |
| `POST /api/places/recommend` | ✅ 基于 TF-IDF + 协同过滤，自动包含新数据 |
| `POST /api/places/sort` | ✅ 通用排序算法 |
| `POST /api/places/{id}/rate` | ✅ 通用评分逻辑 |

### 1.5 前端分析

- 所有页面通过 API 动态加载数据，无硬编码
- `normalize.js` 通用解析 `keywords`/`features` JSON 字段
- 图片路径格式：`uploads/images/XXX.jpg`

### 1.6 测试数据分析

- `src/test/resources/data.sql`：仅 3 条场所（place_campus, place_scenic, place_lake）
- 测试不依赖全部场所数据，不需要扩展

## 三、数据生成规范

### 3.1 字段生成规则

| 字段 | 生成规则 |
|------|----------|
| `id` | `place_202` ~ `place_301`（连续编号） |
| `name` | 真实景区名称 |
| `type` | 固定值 `"景区"` |
| `keywords` | 6个关键词：`["景区名","城市","省","核心特色1","核心特色2","景区等级"]` |
| `features` | 4个特色标签，使用中文四字短语风格 |
| `rating` | 随机 3.5~5.0，保留一位小数 |
| `rating_count` | 随机 5000~100000 整数 |
| `click_count` | 随机 1000~100000 整数 |
| `lat/lng` | 真实经纬度坐标（WGS84，精确到 7 位小数） |
| `address` | 真实地址，格式：`XX省XX市XX区/县XX路XX号` |
| `open_time` | 真实开放时间或 "全天开放" |
| `image` | `uploads/images/XXX.jpg`（编号与场所对应） |
| `description` | 约 80-120 字简介，突出最核心特色 |
| `detail_description` | 约 300-500 字详细介绍，包含地理、历史、文化、游览建议 |

### 3.2 数据质量保证（QA Checklist）

> ⚠️ **关键要求**：所有数据必须通过以下检查，不允许任何缺失或错误。

1. **完整性检查**
   - [ ] 每个字段都有值（NULL 字段必须明确标记）
   - [ ] keywords 至少 4 个元素
   - [ ] features 至少 4 个元素
   - [ ] description 80-120 字
   - [ ] detail_description 300-500 字
   - [ ] lat/lng 为真实坐标

2. **格式一致性检查**
   - [ ] id 格式统一：`place_NNN`（3位数字补零）
   - [ ] JSON 字段使用双引号：`["a","b"]`
   - [ ] rating 格式：`X.X`（1位小数）
   - [ ] 地址格式统一：省级+市级+区级+详细地址

3. **唯一性检查**
   - [ ] id 不与现有数据重复
   - [ ] name 不重复
   - [ ] lat+lng 组合不重复（同一位置）

4. **合理性检查**
   - [ ] lat 范围：18° ~ 54°（中国境内）
   - [ ] lng 范围：73° ~ 135°（中国境内）
   - [ ] rating 范围：0.0 ~ 5.0
   - [ ] 景区名称与坐标匹配（城市级校验）

```sql
-- 验证总数

-- 验证无重复
SELECT id, COUNT(*) FROM spot_place GROUP BY id HAVING COUNT(*) > 1;

-- 验证无 NULL 字段
SELECT id, name FROM spot_place WHERE 
  keywords IS NULL OR features IS NULL OR rating IS NULL 
  OR lat IS NULL OR lng IS NULL OR address IS NULL 
  OR open_time IS NULL OR image IS NULL 
  OR description IS NULL OR detail_description IS NULL;

SELECT id, name, lat, lng FROM spot_place WHERE
```

## 八、风险与注意事项

### 8.1 数据风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 经纬度数据错误 | 地图显示位置偏移 | 通过高德地图 API 批量验证坐标 |
| JSON 格式错误 | 前端解析异常 | 运行时 JSON 校验 + 自动化测试 |
| 景区名称重复 | 搜索混淆 | 编写前交叉比对去重 |
| detail_description 长度不足 | 详情页内容单薄 | 字数统计工具逐条检查 |

### 8.2 性能风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据量增加导致分页查询变慢 | 列表加载变慢 | MySQL 已有 idx_type/idx_rating/idx_click_count 索引，影响可控 |
| 推荐算法计算量增加 | 推荐接口变慢 | 推荐算法已在内存中运行，100→200 条对 TF-IDF 计算增量极小 |

### 8.3 兼容性风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 图片文件不存在 | 封面空白 | 前端应添加默认占位图逻辑（如有必要） |
| 旧缓存数据 | 列表不显示新景区 | 数据更新后清理 Redis 缓存 |

## 十、总结

本次拓展的核心原则：

1. **零代码改动**：系统架构已支持动态扩展，只需追加数据，不需要修改任何 Java/JS/Swift 代码
2. **数据完整性优先**：每个景区 18 个字段必须有值，不允许 NULL 缺失
3. **格式一致性**：新数据必须与现有 place_102~place_201 的风格、格式、数据范围完全一致
4. **地理真实性**：坐标、地址、开放时间必须真实可查
5. **测试驱动验证**：先写验证 SQL，再生成数据，最后启动验证
