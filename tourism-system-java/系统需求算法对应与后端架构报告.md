# 个性化旅游系统需求算法对应与后端架构报告

## 1. 报告依据

本文根据项目需求文档 `requirements_summary_from_images.md` 和当前后端源码整理，重点回答三件事：

1. 每一个功能需求具体对应后端中的哪类算法。
2. 后端整体方案架构中各模块使用了什么技术栈。
3. 汇报时可以如何说明系统的创新点、算法选择和工程落地方式。

当前重点算法代码位于：

```text
src/main/java/com/tourism/algorithm
```

主要算法类包括：

| 算法类 | 主要作用 |
| --- | --- |
| `RecommendationAlgorithm` | 个性化推荐、内容推荐、协同过滤预留、热度推荐、混合推荐 |
| `SortingAlgorithm` | 快速排序、归并排序、堆排序、QuickSelect Top-K、多键加权排序 |
| `SearchAlgorithm` | 二分查找、线性查找、关键词搜索、Jaccard/N-gram 模糊搜索、TF-IDF 全文检索、哈希索引 |
| `ShortestPathAlgorithm` | A*、Dijkstra、混合交通工具 Dijkstra、多目标 TSP 路线规划 |
| `IndoorNavigationAlgorithm` | 室内导航图加载、室内 Dijkstra、楼层切换、电梯/楼梯时间估算 |
| `TreapTopK` | 基于 Treap 的 Top-K 数据结构，用于动态维护推荐前 K 个结果 |
| `HuffmanCompression` | 旅游日记内容 Huffman 无损压缩与解压 |
| `ACAutomaton` | AC 自动机，多模式字符串匹配，当前用于日记敏感词过滤，也可用于多关键词文本匹配 |
| `FibonacciHeap` | 斐波那契堆优先队列实现，适合图算法优化，目前作为算法储备类存在 |

## 2. 创新需求的体现

根据 `requirements_summary_from_images.md`，本系统的创新需求不是单一功能创新，而是体现在“旅游业务场景 + 算法能力 + 多媒体交互”的综合设计上。

### 2.1 双场景智慧旅游

系统同时面向景区和校园两个场景。传统旅游系统一般只服务景区，本系统把校园参观也纳入旅游场景，支持学校、教学楼、博物馆、校园设施等对象的推荐、查询和导航。

创新体现：

- 景区和校园统一建模为“场所”。
- 校园支持步行、自行车等交通方式。
- 景区支持步行、电瓶车等交通方式。
- 室外道路和室内建筑都可以参与导航。

### 2.2 个性化推荐创新

系统不是简单展示热门景点，而是结合用户兴趣、浏览行为、热度和评分进行推荐。

创新体现：

- 景点/学校推荐支持“热度 + 评分 + 兴趣”的综合排序。
- 旅游日记推荐支持根据用户兴趣匹配内容。
- 美食推荐支持按照场所、菜系、热度、评价等维度筛选。
- 用户通常只看前 10 个结果，因此系统使用 Top-K 算法提升推荐效率。

### 2.3 多策略路径规划创新

系统的路线规划不只是求两点之间最短路，而是支持多种真实旅游约束。

创新体现：

- 支持单目标最短路径。
- 支持多目标游览，并要求游览后返回起点。
- 支持最短距离、最短时间两种策略。
- 支持道路拥挤度对通行时间的影响。
- 支持步行、自行车、电瓶车等交通工具限制。
- 支持多种交通工具混合使用。
- 支持室内导航，包括大门、电梯、楼层、房间之间的路径。

### 2.4 真实路径距离查询创新

需求中明确提出，附近设施查询不能只按直线距离排序，而应按可通行路径距离排序。这比普通地图搜索更贴近真实游览体验。

创新体现：

- 查询卫生间、超市等设施时，不只考虑空间位置，还考虑道路是否可达。
- 理论上可通过 Dijkstra 算法计算从当前位置到设施的真实可通行路径距离。
- 查询结果按实际路径距离排序，更符合用户实际行走成本。

当前代码中附近设施主要使用经纬度距离，后续可以把设施节点接入道路图，进一步完整体现该创新需求。

### 2.5 旅游日记交流创新

系统不仅允许用户写日记，还支持日记的推荐、检索、评分、压缩和内容安全处理。

创新体现：

- 日记可以记录文字、图片和视频。
- 日记浏览量作为热度，用户评分作为评价指标。
- 可以按目的地查看相关日记。
- 标题精确查询使用哈希索引。
- 内容检索使用 TF-IDF 全文检索。
- 日记内容使用 Huffman 无损压缩存储。
- 发布内容前使用 AC 自动机进行敏感词过滤。

### 2.6 AIGC 多媒体创新

需求中提出根据景点或学校照片、文字描述生成旅游动画，这是系统区别于普通旅游管理系统的重要创新点。

创新体现：

- 用户上传旅游图片后，系统可以生成动画内容。
- AIGC 模块增强了旅游日记和景点展示的表现形式。
- 后端通过 `AigcService` 调用 Python 脚本完成动画生成，便于后续替换为更强的图像生成或视频生成模型。

### 2.7 汇报总结话术

可以在汇报中这样概括创新需求：

> 本系统的创新点主要体现在三方面：第一，系统融合景区和校园两个旅游场景，实现室内外一体化导航；第二，系统把推荐算法、Top-K 排序、Dijkstra、A*、TSP、TF-IDF、Huffman 和 AC 自动机等算法应用到具体旅游需求中；第三，系统加入旅游日记交流和 AIGC 动画生成，使系统从“查景点、看路线”扩展为“推荐、导航、记录、交流、生成内容”的综合智慧旅游平台。

## 3. 需求与具体算法对应表

### 3.1 旅游推荐

| 需求点 | 对应后端模块 | 对应算法/技术 | 对应代码 |
| --- | --- | --- | --- |
| 用户根据喜好选择景点和学校作为旅游目的地 | 场所模块、用户模块 | 用户兴趣标签解析、用户画像构建 | `PlaceController`、`RecommendationAlgorithm.buildUserProfile` |
| 浏览前推荐旅游景点和学校 | 场所推荐模块 | 混合推荐：内容推荐 + 热度推荐；无兴趣时退化为热度推荐 | `RecommendationAlgorithm.recommendPlaces`、`hybridRecommendation`、`popularityBasedRecommendation` |
| 按旅游热度、评价和个人兴趣推荐 | 推荐模块、排序模块 | 评分、点击量、兴趣匹配加权打分；余弦相似度；Top-K | `RecommendationAlgorithm.buildItemProfile`、`cosineSimilarity`、`TreapTopK` |
| 输入名称、类别、关键字查询景点/学校 | 场所查询模块 | 数据库条件查询、LIKE 模糊匹配、分类过滤 | `PlaceServiceImpl.listPlaces`、`PlaceController.search` |
| 多项查询结果按热度和评价排序 | 场所排序模块 | Top-K 排序、快速排序/QuickSelect 自适应选择 | `PlaceController.sort`、`SortingAlgorithm.topKSort` |
| 用户通常只看前 10 个，要求不完全排序得到前 10 个 | 推荐模块、排序模块 | QuickSelect Top-K、堆 Top-K、Treap Top-K | `SortingAlgorithm.quickSelectTopK`、`heapSortTopK`、`TreapTopK` |
| 数据动态变化时仍能维护前 K 个推荐 | 推荐模块 | Treap 动态插入和淘汰最低分项目 | `TreapTopK.insert`、`getTopK` |

推荐汇报说法：

> 旅游推荐不是简单按数据库排序，而是把用户兴趣、历史浏览、场所标签、热度和评分组合成综合分。为了满足用户只关注前 10 个结果的场景，系统使用 Top-K 算法，避免对全量数据完整排序。

### 3.2 旅游路线规划

| 需求点 | 对应后端模块 | 对应算法/技术 | 对应代码 |
| --- | --- | --- | --- |
| 单目标路线规划：从当前位置到目标景点/场所 | 导航模块 | A* 最短路径；距离策略可用 Dijkstra | `NavigationController.shortestPath`、`ShortestPathAlgorithm.astar`、`dijkstraDistanceOnly` |
| 以距离最短为目标 | 导航模块 | Dijkstra，边权为道路距离 | `ShortestPathAlgorithm.dijkstraDistanceOnly` |
| 以时间最短为目标 | 导航模块 | A*，边权为通行时间；启发函数使用经纬度距离估算 | `ShortestPathAlgorithm.astar`、`heuristic` |
| 考虑道路拥挤度 | 导航模块 | 真实速度 = 理想速度 × 拥挤度 × 交通工具速度系数；时间作为边权 | `ShortestPathAlgorithm.calculateTravelTime` |
| 校园内步行/自行车 | 导航模块 | 交通工具约束边过滤；校园可用 `步行`、`自行车` | `ShortestPathAlgorithm.getAvailableVehicles`、`calculateTravelTime` |
| 景区内步行/电瓶车 | 导航模块 | 交通工具约束边过滤；景区可用 `步行`、`电瓶车` | `ShortestPathAlgorithm.getAvailableVehicles`、`calculateTravelTime` |
| 不同交通工具可组合使用 | 导航模块 | 状态扩展 Dijkstra：状态为 `[节点, 交通工具]` | `ShortestPathAlgorithm.dijkstraWithMixedVehicles` |
| 多目标路线规划，游览多个点后返回起点 | 导航模块 | TSP 路线规划；两两距离用 Dijkstra 预计算 | `NavigationController.multiDestination`、`ShortestPathAlgorithm.multiDestinationRoute` |
| 多目标点较少时求更优路线 | 导航模块 | 动态规划 TSP，适合目标点数量不超过 15 | `ShortestPathAlgorithm.tspDynamicProgrammingDistanceOnly` |
| 多目标点较多时近似优化 | 导航模块 | 最近邻、模拟退火、遗传算法 | `tspNearestNeighborDistanceOnly`、`tspSimulatedAnnealingDistanceOnly`、`tspGeneticAlgorithmDistanceOnly` |
| 路线图形化展示 | 前端地图展示 + 后端路径接口 | 后端返回路径节点、分段距离、分段时间，前端负责地图绘制 | `NavigationController.shortestPath`、`mixedVehiclePath`、`multiDestination` |
| 室内导航：大门到电梯、楼层间、楼层内到房间 | 室内导航模块 | 室内图建模 + Dijkstra；边权支持距离/时间；电梯和楼梯楼层切换时间估算 | `IndoorNavigationAlgorithm.navigateToRoom`、`dijkstra`、`estimateTimeWithFloorChanges` |

路线规划汇报说法：

> 系统将景区道路、校园道路和室内结构都抽象成图。节点表示景点、建筑、路口、房间、电梯等，边表示可通行道路或室内连接。单目标路径使用 Dijkstra/A*，多目标游览使用 TSP 系列算法，混合交通工具通过“节点 + 交通工具”的扩展状态图实现。

### 3.3 场所查询与附近设施

| 需求点 | 对应后端模块 | 对应算法/技术 | 对应代码 |
| --- | --- | --- | --- |
| 选中某地点后查询附近超市、卫生间等设施 | 设施模块 | 当前实现为经纬度 LBS 半径过滤 | `FacilityController.nearby`、`GeoUtils.distance` |
| 附近设施按距离排序 | 设施模块 | 当前 `/nearest` 使用经纬度直线距离排序；算法包中也有 Haversine 排序方法 | `FacilityController.nearest`、`SortingAlgorithm.sortFacilitiesByDistance` |
| 距离不能按直线距离，应基于可通行路径距离 | 导航模块 + 设施模块 | 应使用 Dijkstra 计算从当前节点到设施节点的可通行路径距离，再排序 | 建议接入 `ShortestPathAlgorithm.dijkstraDistanceOnly` |

> 需求文档明确要求“附近设施排序不能依据直线距离，应基于可通行路径距离”。当前代码里的附近设施接口主要使用 `GeoUtils.distance` 计算经纬度直线距离，和需求还有差距。汇报时可以说明当前已具备 Dijkstra 路径算法能力，后续可以把设施节点接入道路图，用 `ShortestPathAlgorithm.dijkstraDistanceOnly` 替换直线距离排序，从而完全满足需求。

## 4. 算法类与需求关系总览

| 算法类 | 对应需求 |
| --- | --- |
| `RecommendationAlgorithm` | 景点/学校推荐、旅游日记推荐、热度推荐、个性化推荐 |
| `SortingAlgorithm` | 推荐结果排序、热度排序、评分排序、Top-K 场景、美食排序、日记相关度排序 |
| `SearchAlgorithm` | 景点/学校关键字查找、美食模糊查询、设施类别查找、日记标题查找、日记内容全文检索 |
| `ShortestPathAlgorithm` | 单目标路线规划、最短距离、最短时间、拥挤度、交通工具约束、多目标路线规划 |
| `IndoorNavigationAlgorithm` | 教学楼/博物馆等建筑内部导航、楼层切换、电梯导航、房间导航 |
| `TreapTopK` | 动态 Top-K 推荐、推荐结果实时维护 |
| `HuffmanCompression` | 旅游日记无损压缩存储 |
| `ACAutomaton` | 日记敏感词过滤、多关键词文本匹配 |
| `FibonacciHeap` | 图算法优先队列优化储备，可用于优化 Dijkstra/A* 的 decreaseKey 性能 |

## 5. 后端整体方案架构与技术栈

### 5.1 总体分层架构


## 6. 汇报可用的整体方案表述

系统整体采用“业务服务 + 算法引擎 + 数据支撑”的设计：

1. 业务服务层负责旅游推荐、路线规划、设施查询、日记交流、美食推荐和媒体上传等功能。
2. 算法引擎层负责推荐排序、Top-K、搜索、图路径规划、TSP 多点游览、室内导航、Huffman 压缩和 AC 自动机文本匹配。
3. 数据支撑层使用 MySQL 存储用户、场所、道路、日记、设施、美食和行为数据；使用道路拓扑表支持路径规划；使用用户行为表支持个性化推荐。
4. 安全与接口层使用 Spring Security + JWT 实现登录鉴权，使用 Swagger/OpenAPI 方便接口测试和汇报展示。
5. AIGC 模块通过 Java 后端调用 Python 脚本，将旅游图片合成为动画，作为系统多媒体创新点。

## 7. 系统创新点总结

1. **推荐算法与旅游业务结合**：系统不是固定展示景点，而是根据用户兴趣、评分、热度、历史行为计算推荐结果。
2. **Top-K 优化**：针对用户只看前 10 个结果的场景，使用 QuickSelect、堆和 Treap 维护 Top-K，提高排序效率。
3. **多策略路径规划**：支持最短距离、最短时间、拥挤度、交通工具约束和混合交通工具路线。
4. **多目标游览路线**：将多个景点游览并返回起点的问题建模为 TSP，支持最近邻、动态规划、模拟退火和遗传算法。
5. **室内外一体导航**：室外用道路拓扑图，室内用建筑 JSON 图，支持电梯、楼梯和楼层切换时间估算。
6. **日记交流具备算法能力**：标题哈希索引、TF-IDF 全文检索、Huffman 压缩、AC 自动机过滤，使日记模块具备搜索、存储和安全能力。
7. **美食推荐与模糊搜索**：支持按景点/学校关联美食，并用 Jaccard/N-gram 相似度实现模糊查找。
8. **AIGC 动画扩展**：支持将旅游图片生成动画，增强旅游日记和景点展示的表现形式。

## 8. 当前实现与需求差距

| 需求 | 当前情况 | 建议 |
| --- | --- | --- |
| 附近设施应按可通行路径距离排序 | 当前主要按经纬度直线距离计算 | 将设施节点接入 `spot_road_edge` 图，调用 `ShortestPathAlgorithm.dijkstraDistanceOnly` 计算路径距离后排序 |
| 美食按距离排序 | 当前主要支持按热度和名称排序，模糊搜索已实现 | 给美食增加坐标或关联设施/餐厅节点，再接入距离排序或路径距离排序 |
| 协同过滤推荐 | `RecommendationAlgorithm` 有协同过滤方法，但业务接口主要使用内容 + 热度混合推荐 | 后续可利用 `user_behavior` 表构建用户相似度，启用协同过滤结果 |
| JGraphT 依赖 | 依赖已引入，但当前核心路径算法为自实现 | 汇报时可说明项目具备引入成熟图算法库的扩展条件 |
| FibonacciHeap | 已实现但当前 Dijkstra/A* 使用 Java `PriorityQueue` | 如果数据规模增大，可用斐波那契堆优化 decreaseKey 场景 |

## 9. 汇报总结

本系统后端以 Spring Boot 为主体，使用 MyBatis-Plus 和 MySQL 管理业务数据，使用 Spring Security + JWT 完成认证授权，并把推荐、排序、搜索、路径规划、压缩和文本匹配等算法封装在独立的 `algorithm` 包中。

从需求对应关系看，旅游推荐主要对应推荐算法和 Top-K 排序；路线规划主要对应 A*、Dijkstra 和 TSP；场所与美食查询主要对应模糊搜索、分类过滤和排序；旅游日记交流主要对应哈希索引、TF-IDF、Huffman 压缩和 AC 自动机；AIGC 动画生成通过后端调用 Python 脚本完成。

因此，本系统可以汇报为一个融合了推荐系统、图算法导航、全文检索、压缩存储、多媒体处理和 Web 后端工程技术的综合型智慧旅游系统。
