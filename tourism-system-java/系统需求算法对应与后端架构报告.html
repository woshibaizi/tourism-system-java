System Report

个性化旅游系统需求、算法与后端架构汇报

面向景区和校园双场景，覆盖 Web 与客户端 App，并集成多 Agent 工程协作能力，将推荐、路径规划、全文检索与压缩存储组合成完整的智慧旅游服务。

2 景区 + 校园双场景
App 移动客户端入口
Agent 工程协作 + 日记生成
Top-K 推荐与排序优化

Innovation

创新需求的体现

创新不只来自某一个功能，而是来自旅游业务场景、算法能力、多端客户端和 Agent 协作机制的组合。

客户端 App 开发

系统不只停留在网页端，还实现了客户端 App 工程，适合移动场景下的路线导航、附近设施查询、日记记录和旅游内容查看。

集成 SystemCraft Agent

集成多 Agent 工程协作能力，通过 PM、架构、开发、测试、导师和学习 Agent 串联需求分析、架构设计、开发实现、测试回流与复盘总结。

旅游日记生成 Agent

集成基于 OpenClaw 搭建的旅游日记 Agent，接入微信后可一键生成旅游日记，并推送到小红书和公众号。

双场景智慧旅游

景区和校园统一建模为场所，校园支持步行、自行车，景区支持步行、电瓶车，室外道路和室内建筑共同参与导航。

个性化推荐

把用户兴趣、浏览行为、热度和评分纳入推荐逻辑，用内容推荐、热度推荐和 Top-K 选择呈现更相关的结果。

多策略路径规划

支持最短距离、最短时间、拥挤度、交通工具约束、多目标返回起点和室内楼层导航。

Architecture

整体方案架构

系统采用前后端分离，后端以 Spring Boot 分层架构组织业务、算法和数据访问。

从场景到算法，再到可交互服务 React 前端发起请求，Spring Boot 后端组织业务，算法层完成推荐、导航、搜索与压缩，数据层提供持久化支撑。
React 前端页面展示、用户交互、地图、图表、日记和媒体上传入口。
Controller提供 REST API，统一接收前端请求并返回 JSON 结果。
Service处理旅游推荐、路线规划、日记交流、美食查询等业务逻辑。
Algorithm封装 Top-K、Dijkstra、A*、TSP、TF-IDF、Huffman、AC 自动机。
Agent集成 SystemCraft 多 Agent 协作与 OpenClaw 旅游日记生成 Agent，覆盖工程流程和内容发布。
Mapper通过 MyBatis-Plus 完成 CRUD、分页、条件查询和逻辑删除。
Data / ExtendMySQL、Redis 和文件系统共同支撑数据存储、缓存与媒体资源管理。

Risks

当前实现与需求差距

汇报时可以主动说明这些点，体现对系统边界和后续优化方向的掌握。

需求当前情况后续建议
附近设施按可通行路径距离排序当前主要按经纬度直线距离计算设施节点接入道路图，用 Dijkstra 计算路径距离后排序
美食按距离排序当前主要支持热度和名称排序给美食增加坐标或餐厅节点，再接入距离排序
协同过滤推荐算法类已有方法，业务接口主要用内容 + 热度利用 user_behavior 表构建用户相似度
图算法库JGraphT 依赖已引入，核心算法当前自实现后续可用 JGraphT 增强图算法能力

本系统融合景区和校园两个旅游场景，实现 Web 与客户端 App 多端使用，并集成 SystemCraft 多 Agent 协作能力和 OpenClaw 旅游日记生成 Agent，将推荐算法、Top-K、Dijkstra、A*、TSP、TF-IDF、Huffman 和 AC 自动机应用到真实业务需求中。
