# 个性化旅游系统 Java 重构开发计划书

本项目旨在将原基于 Python (Flask) 的个性化旅游系统后端重构为基于 **Java (Spring Boot) + MySQL**，前端继续延用保持或升级 **React**。

## 1. 技术选型与架构设计

### 1.1 前端技术栈 (继承与优化)
- **框架**: React 18 + UI 组件库 (Ant Design 或 MUI)
- **路由**: React Router 6
- **状态管理**: Redux Toolkit 或 Zustand
- **地图组件**: Leaflet (配合 react-leaflet) 或百度/高德地图 API
- **请求客户端**: Axios

### 1.2 后端技术栈 (重构核心)
- **核心框架**: Java 17 + Spring Boot 3.x
- **持久层框架**: MyBatis-Plus 或 Spring Data JPA
- **数据库**: MySQL 8.0 (替代原有的 JSON 文件存储，实现高并发和事务支持)
- **缓存/中间件**: Redis (用于热点数据缓存和点赞、排行榜功能)
- **接口文档**: Knife4j / Swagger3
- **安全验证**: Spring Security + JWT 机制

### 1.3 数据规模要求 (严格遵循)
- 系统用户数：$\ge$ 10 人 (需准备初始化 SQL 脚本)
- 景区和校园数量：$\ge$ 200 个
- 建筑物数量：$\ge$ 20 个 (含景点、教学楼、办公楼、宿舍楼等)
- 其他服务设施：$\ge$ 10 种 (商店、饭店、洗手间、图书馆、食堂、超市、咖啡馆等)，总数 $\ge$ 50 个
- 边数(路网数据)：$\ge$ 200 条边 (可通过爬取开源地图如 OpenStreetMap 数据转换，或自主编写连通图生成脚本)

## 2. 数据库设计 (MySQL)

原系统数据以 `.json` 静态文件存储。针对 Java+MySQL 架构，需要设计以下核心物理表：

1. **`sys_user` (用户表)**：用户基本信息，包含偏好标签(兴趣)。
2. **`spot_place` (场所表)**：记录 200+ 个景区与校园实体，包含总热度、全局评分、经纬度。
3. **`spot_building` (建筑物表)**：隶属于具体的 `place_id`，记录主要建筑。
4. **`spot_facility` (服务设施表)**：隶属于具体的 `place_id`，记录分类字典(10种以上)与位置。
5. **`spot_road_edge` (道路拓扑表)**：记录 `source_id`, `target_id`, `distance`, `transport_type` 等，用于图算法。
6. **`travel_diary` (旅游日记表)**：用户发表的旅游图文、归属的场所。
7. **`user_behavior` (用户行为表)**：记录用户的浏览、点赞、评分记录，为协同过滤算法提供数据源。

## 3. 核心功能及实现方案

结合旅游的三个阶段，具体功能实现细节如下：

### 3.1 旅游前：目的地智能推荐
- **功能描述**: 用户登录后，系统根据热度、评价和个人兴趣推荐旅游场所(景区/校园)。
- **Java 实现**:
  1. **热度与评价统计**: 利用 MySQL `GROUP BY` 和 Redis 的 `ZSET` 进行排行榜计算与缓存。
  2. **基于内容的推荐**: 根据用户画像中的兴趣标签，通过 Elasticsearch 或 MySQL 的全文检索去匹配景点的 tag。
  3. **协同过滤推荐**: 将原 Python 的协同过滤算法转译为 Java 侧实现（可用 Apache Mahout 库，或直接在 Java 服务内编写 User-CF/Item-CF 内存级计算逻辑：定时计算用户相似度矩阵并写入 Redis 缓存）。

### 3.2 旅游中：路线规划与场所查询
- **功能描述**: 进入指定景区或校园后，根据目标规划最优路线，同时支持周边服务设施查询。
- **Java 实现**:
  1. **最优路径规划 (单目标/两点间)**:
     - 将关系数据库中的 `spot_road_edge` 表加载到内存中，构建有向带权图（比如使用 `JGraphT` 库，或者手写邻接表）。
     - 实现 **A* 算法** 或 **Dijkstra 算法** 计算两点间最短路径，以提供高效率服务。
  2. **景区参观线路规划 (多目标 TSP)**:
     - 采用 **TSP（旅行商问题）** 近似解算法。在 Java 中可实现动态规划（针对目标较少），或使用蚁群算法 / 遗传算法工具包进行多点途经的最优路线计算。
  3. **场所与设施查询 (LBS / 空间搜索)**:
     - 结合 MySQL 8.0 的空间数据类型 (Spatial Data / `POINT`) 进行距离计算计算。通过 ST_Distance_Sphere 函数快速获得 500m 内的洗手间、咖啡馆等设施。
  4. **知识拓展与介绍**: 结合大模型 API (如调用通义千问 / Claude / ChatGPT 的 Java SDK) 进行景点的动态介绍生成，或直接返回后台配置的介绍文本。

### 3.3 旅游后：日记生成与交流区
- **功能描述**: 用户上传照片，系统结合经历生成图文并茂的旅游日记，供大家交流评价。
- **Java 实现**:
  1. **多媒体上传**: 使用 Spring Boot 集成阿里云 OSS、腾讯云 COS 或本地 MinIO，提供图片和短视频的上传接口，存储对象 URL 保存到 MySQL 中。
  2. **AIGC 日记辅助**: 当用户提交几张照片或几个关键词时，后台调用 LLM 的 API 接口，返回一段润色过的"旅游日记草稿"。
  3. **日记展示与互动**: 实现分页查询（PageHelper / Mybatis-Plus）、点赞接口（利用 Redis `Set` 防止重复点赞）、评论嵌套表。

## 4. 后台代码结构规约

建议采用标准的 DDD (领域驱动设计) 或 MVC 分层结构：
```text
tourism-system-java/
├── config/              # Redis, Security, Swagger 等配置类
├── controller/          # 表现层：定义 RESTful API
├── service/             # 业务逻辑层接口
│   └── impl/            # 业务逻辑层实现 (算法调用主要在此处)
├── mapper/              # 数据持久层 (MyBatis)
├── model/
│   ├── entity/          # 对应数据库表
│   ├── dto/             # 前后端交互的数据传输对象
│   └── vo/              # 视图对象
├── algorithm/           # 移植 Python 的核心算法 (A*, TSP, CF等)
├── exception/           # 全局异常处理
└── utils/               # 工具类 (JWT, 距离计算, 图形转化)
```

## 5. 项目推进路线图 (Roadmap)

1. **第一阶段：需求对接与环境搭建 (Day 1 - 2)**
   - 梳理 JSON 数据，编写 Python 或 Java 抽取脚本，将 `places.json`, `roads.json` 数据洗入 MySQL 数据库。
   - 搭建 Spring Boot 工程骨架，集成基础中间件，打通前端 React 联调链路。

2. **第二阶段：基础服务与CRUD实现 (Day 3 - 5)**
   - 实现用户登录、注册 (Spring Security + JWT)。
   - 实现场所的列表分页、搜索、展示服务。
   - 实现多媒体游记论坛板块。

3. **第三阶段：算法移植与对接 (Day 6 - 9)**
   - 将原 Python 的 A* 和 TSP 逻辑使用 Java 重写，进行测试覆盖，确保返回的路径节点能被前端地图（Leaflet）读取并绘制高亮线条。
   - 编写推荐算法的定时刷新任务。

4. **第四阶段：系统联调、压测与部署 (Day 10 - 12)**
   - 使用 JMeter 等工具模拟 $\ge$ 10 个用户并发访问。
   - 完善数据，确保达到 "200+景点、200+边" 的要求。

## 6. 详细开发流程与优先级 (建议开发顺序)

为了保证项目的顺利推进，建议采用**自底向上构建，按核心功能逐渐向外围扩展**的开发策略。具体流程如下：

### 第一步：环境搭建与数据准备 (优先级：最高)
*这是整个项目的基础，必须优先完成。*
1. **数据库设计与创建**：根据前文的数据库设计，在 MySQL 中建表（`sys_user`, `spot_place`, `spot_building`, `spot_facility` 等）。
2. **数据迁移脚本**：编写 Python 或 Java 抽取脚本，将原项目 `backend/data/` 下的 JSON 数据解析并插入到 MySQL 数据库中。
3. **SpringBoot 工程初始化**：使用 Spring Initializr 搭建基础框架，集成 MyBatis-Plus、Redis、Swagger 等基础设施。

### 第二步：基础 CRUD 与用户系统 (优先级：高)
*构建系统的骨架，让前后端能够跑通。*
1. **用户认证模块**：实现基于 Spring Security + JWT 的登录、注册、用户信息获取接口。
2. **场所基础信息管理**：实现对旅游目的地（景区/校园）、建筑物、服务设施的分页查询和详情接口。
3. **前端对接测试**：让 React 前端能够成功调用上述基础接口并展示数据。

### 第三步：核心算法与旅游中功能 (优先级：高，项目难点)
*旅游路线规划是本系统的核心需求，需要在业务逻辑层重点攻克。*
1. **路网数据加载**：将 `spot_road_edge` 数据加载到 Java 内存中（可利用 `JGraphT` 库或自定义数据结构），构建图结构。
2. **单点路径规划 (A* 算法)**：Java 重写原 Python 中的 A* 算法，实现同一景区内两个建筑物/设施间的最短路径计算。
3. **多点 TSP 规划**：Java 实现 TSP 近似算法，解决游客在景区内的完整游览路线规划。
4. **空间查询 (LBS)**：利用 MySQL 8 空间函数或经纬度计算公式，实现特定半径内的服务设施查询。

### 第四步：旅游前与旅游后功能 (优先级：中)
*增强业务完整性与用户体验。*
1. **智能推荐模块 (旅游前)**：
   - 结合 MySQL/Redis 实现热门推荐排行。
   - 移植内容推荐和协同过滤算法逻辑至 Java 后端。
2. **多媒体与旅游日记 (旅游后)**：
   - 集成 MinIO 或云存储对象服务，实现图片、视频上传接口。
   - 实现日记的发布、查询、点赞、评论等互动功能。
   - (可选) 接入 LLM API，实现 "一键生成日记" 的 AIGC 辅助功能。

### 第五步：联调、测试与优化 (优先级：收尾阶段)
1. **端到端测试**：前后端联合测试所有业务主流程（登录 -> 推荐 -> 规划 -> 游记）。
2. **性能与并发测试**：使用 JMeter 进行超过 10 人并发的压力测试。
3. **缓存优化**：利用 Redis 优化高频查询（如热门景点列表），提升系统吞吐量。

**附录: 算法迁移贴士**
原仓库下的 `backend/algorithms/*.py` (如 `shortest_path_algorithm.py`) 可以借助大语言模型批量翻译为 Java 版本并封装为独立的服务类注入 Spring Bean，通过单例模式保持常驻内存的图结构，避免每次算法请求时重复从数据库查询边集导致效率低下。
